# AGENT.md

This file is for humans and coding agents working inside this repository. It describes the current code shape, the project’s operating assumptions, and the conventions that matter when making changes.

## Project Summary

`promptd` is a single-binary Go application with an embedded React frontend.

Primary capabilities:

- multi-provider OpenAI-compatible chat
- JWT cookie auth and bearer service tokens
- RBAC for permissions and allow-listed models/tools/system prompts
- MCP tool integration
- YAML persistence for conversations, uploads, schedules, and executions
- rolling conversation compaction
- scheduler with cron and one-time jobs
- detailed trace capture and frontend trace viewers
- optional HTTPS with provided or autogenerated certificates

## Runtime Model

Startup sequence in broad terms:

1. load YAML config
2. create logger
3. validate provider config
4. build built-in tool registry
5. start MCP manager
6. load system prompts
7. open YAML storage
8. create auth service and session store
9. build provider registry
10. construct handlers
11. start scheduler
12. register HTTP routes
13. start HTTP or HTTPS server

## Current Auth And RBAC Behavior

- Auth supports JWT session cookies and bearer service tokens.
- Users are tenant scoped.
- Roles merge by union.
- `super_admin` bypasses model/tool/prompt allow lists and receives full boolean permissions.
- Boolean permissions are enforced server-side.
- Allow lists are enforced both when listing options and when executing requests.

Current permission flags:

- `chat`
- `upload`
- `conversations_read`
- `conversations_write`
- `compact_conversation_write`
- `schedules_read`
- `schedules_write`
- `traces_read`
- `admin`

## Current Chat Behavior

- System prompt is required.
- Provider can be selected independently from model.
- If model is omitted, backend chooses from allowed candidates.
- Chat supports file uploads.
- Tool calling uses built-in and MCP tools filtered through RBAC.
- Intermediate tool-loop messages are transient and not persisted.
- Assistant traces are persisted and later pruned by TTL.
- Conversation edit flow truncates from the edited message onward and then resends.

## Current Compaction Behavior

- Auto compaction is controlled by `llm.compact_conversation`.
- Manual compaction is allowed only with `compact_conversation_write`.
- A rolling compact summary is persisted as a special assistant message.
- Future LLM calls use the summary plus newer messages after the compaction cutoff.
- Compaction itself still respects allowed provider/model rules.

## Current Scheduler Behavior

- Supports `cron` and `once` schedules.
- Rebuilds principal from stored tenant/user scope at execution time.
- Re-checks prompt/model/tool permissions during execution.
- Saves executions with response, trace, token usage, and model/provider used.
- One-time schedules disable themselves after a run.
- History retention is enforced by deleting oldest execution files when necessary.

## Current Frontend Behavior

- Main views are shell-managed through pathname parsing, not a React Router route tree.
- Chat remembers selected provider/model/system prompt in local storage.
- Conversation history supports search, pin, rename, delete.
- Scheduler supports list, form, and execution history/detail views.
- Markdown rendering is customized in `web/src/components/markdown/`.
- Trace rendering is shared between chat bubbles and scheduler executions.

## Current Data Layout

Data is stored under:

```text
<data.dir>/tenants/<tenant>/users/<user>/
```

Within that scope:

- `conversations/`
- `uploads/`
- `schedules/`
- `schedules/executions/`

TLS autogenerated certs are stored at:

```text
<data.dir>/tls/
```

## Important Constraints

- Keep config-driven behavior in config, not hardcoded per user.
- Preserve RBAC filtering and validation on the backend.
- Do not make the frontend silently choose models when the backend already owns fallback selection.
- Keep chat/system prompt validation strict.
- System prompt definitions live under `llm.system_prompts`.
- App branding is built-in; keep it code-owned unless there is a concrete product need for configurable branding.
- Avoid broad global CSS fixes when a markdown-specific or component-specific fix is enough.
- Prefer minimal changes over introducing new abstractions without clear reuse.

## Known Caveats

- Session cookies currently use `Secure: false` even when TLS is enabled.
- MCP tool results are effectively text-oriented.
- There are currently no meaningful Go tests in the repository.
- `config.yaml` in a working tree may contain real-looking secrets; do not rely on it as safe sample data.
